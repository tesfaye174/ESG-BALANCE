Cos'e' ESG-BALANCE

E' un'applicazione web che permette di gestire i bilanci di esercizio delle aziende e i relativi indicatori ESG (Environmental, Social, Governance). In pratica il sistema gestisce tutto il ciclo di vita di un bilancio: dalla creazione alla revisione fino all'approvazione o al respingimento.

L'idea nasce dal fatto che oggi le aziende devono rendicontare non solo i dati finanziari ma anche l'impatto ambientale, sociale e di governance. Ho voluto creare un sistema che integra questi due aspetti in un unico flusso di lavoro, con un workflow di revisione completo (bozza → in revisione → approvato/respinto).

I tre ruoli del sistema

Ruolo Cosa fa
Amministratore Gestisce il template contabile, definisce gli indicatori ESG (ambientali, sociali, governance) e assegna i revisori
Responsabile Registra le aziende, crea i bilanci, compila i valori delle voci contabili e collega gli indicatori ESG ai dati
Revisore ESG Dichiara le competenze (livello 0-5), revisiona i bilanci con note puntuali e alla fine emette un giudizio finale

Tecnologie usate

  • Backend: PHP 8 procedurale, con PDO e prepared statements
  • Database: MySQL/MariaDB — 17 tabelle in BCNF
  • Frontend: HTML5, Bootstrap 5.3, JavaScript vanilla
  • Server: Apache tramite XAMPP
  • Sicurezza: Bcrypt (cost 12), rigenerazione sessione, htmlspecialchars su tutti gli output

Funzionalita' principali

  • Dashboard personalizzata per ruolo
  • 4 pagine statistiche con viste SQL
  • Log eventi su tabella MySQL dedicata
  • Upload file (loghi aziende, CV revisori)

Schema E-R (Progettazione Concettuale)

Gerarchie

  • UTENTE → Revisore / Responsabile
    Totale, Esclusiva (discriminatore: ruolo). L'amministratore non ha sotto-tabella dedicata, è solo un valore dell'ENUM.
  • INDICATORE_ESG → Ambientale / Sociale
    Parziale, Esclusiva (discriminatore: tipo). Gli indicatori di governance restano solo nella tabella padre.

Entita' principali (con identificatori)

  • UTENTE — id: username (interno)
  • AZIENDA — id: codice auto-incrementale (interno), ragione_sociale (esterno UNIQUE)
  • VOCE_CONTABILE — id: nome (interno)
  • BILANCIO — id: codice auto-incrementale (interno), stato ENUM con 4 valori
  • INDICATORE_ESG — id: nome (interno), rilevanza CHECK 0-10, tipo ENUM
  • EMAIL — attributo multivalore di UTENTE
  • NOTA_REVISIONE — id: codice auto-incrementale
  • GIUDIZIO — id: (username_revisore, id_bilancio) composito

Relazioni (con cardinalita')

  • GESTISCE — Responsabile (1,N) : Azienda (1,1)
    Un responsabile gestisce 1+ aziende, ogni azienda ha esattamente 1 responsabile
  • APPARTIENE — Azienda (0,N) : Bilancio (1,1)
    Un'azienda puo' avere 0+ bilanci, ogni bilancio appartiene a 1 azienda
  • VALORIZZA — Bilancio : Voce (M:N)
    Con attributo di relazione: valore DECIMAL(15,2)
  • COLLEGA — relazione ternaria Bilancio-Voce-Indicatore
    Con attributi: valore_indicatore, fonte, data_rilevazione
  • REVISIONA — Revisore (0,N) : Bilancio (0,N)
    Assegnazione M:N, tabella di raccordo
  • GIUDICA — Revisore (0,1) : Bilancio (0,N)
    Max 1 giudizio per revisore per bilancio (PK composita)

Le 14 Stored Procedure

Ho deciso di fare passare tutte le operazioni di scrittura tramite stored procedure, cosi' la logica sta dentro il database e non nel codice PHP. In questo modo il codice applicativo chiama solo CALL sp_nome(...) e non fa mai INSERT/UPDATE direttamente. Questo e' utile anche per la manutenzione: se devo cambiare la logica di business, modifico solo la SP senza toccare il PHP.

# Nome Cosa fa
SP1 sp_login Ritorna username, hash password e ruolo
SP2 sp_registra_utente Registra utente + email + sotto-tabella in modo atomico
SP3 sp_aggiungi_email Aggiunge un'email a un utente
SP4 sp_crea_voce_contabile Crea una voce nel template contabile
SP5 sp_inserisci_indicatore_esg Inserisce indicatore + eventuale sotto-tabella (ambientale/sociale/governance)
SP6 sp_registra_azienda Registra un'azienda per il responsabile
SP7 sp_crea_bilancio Crea bilancio in stato 'bozza', ritorna l'ID
SP8 sp_inserisci_valore_bilancio UPSERT del valore di una voce (ON DUPLICATE KEY UPDATE)
SP9 sp_collega_indicatore_voce Collega un indicatore ESG a una voce del bilancio
SP10 sp_associa_revisore_bilancio Assegna revisore al bilancio, incrementa nr_revisioni
SP11 sp_inserisci_competenza UPSERT competenza del revisore
SP12 sp_inserisci_nota Inserisce nota del revisore su una voce
SP13 sp_inserisci_giudizio Registra il giudizio finale del revisore

Ciclo di vita del bilancio

v_affidabilita_aziende: calcola per ogni azienda la % di bilanci approvati senza rilievi (approvazione pura).
v_classifica_bilanci_esg: ordina i bilanci per numero di indicatori ESG collegati, utile per capire quali bilanci sono piu' completi.

Normalizzazione (BCNF)

Ho verificato che tutte le 17 tabelle sono in BCNF: per ogni dipendenza funzionale non banale X → Y, X e' una superchiave. Queste sono alcune delle verifiche che ho fatto:

Tabella Chiave Dipendenze funzionali OK?
utenti {username} username → password_hash, CF, data_nascita, luogo_nascita, ruolo Si'
email_utente {username, email} Nessun attributo non-chiave Si'
aziende {id}, {ragione_sociale} Entrambi sono superchiavi → ok Si'
valori_bilancio {id_bilancio, nome_voce} PK → valore Si'
voci_indicatori {id_bilancio, nome_voce, nome_indicatore} PK → valore_indicatore, fonte, data Si'
giudizi {username_revisore, id_bilancio} PK → esito, data_giudizio, rilievi Si'

Come ho ristrutturato lo schema E-R

  • Gerarchia utenti (Totale, Escl.): tabella padre utenti + sotto-tabelle revisori e responsabili con stessa PK
  • Gerarchia indicatori (Parz., Escl.): tabella padre + sotto-tabelle indicatori_ambientali e indicatori_sociali
  • Email multivalore: ho messo una tabella separata con PK composita (username, email)
  • Relazione M:N (VALORIZZA): tabella valori_bilancio con PK composita
  • Relazione ternaria (COLLEGA): tabella voci_indicatori con PK su 3 attributi

Codice SQL: Trigger T2 + SP14

Questi sono i due pezzi di codice piu' importanti del progetto: il trigger che decide lo stato finale del bilancio e la SP che ricalcola l'affidabilita' del revisore.

Trigger T2 — trg_bilancio_giudizio

CREATE TRIGGER trg_bilancio_giudizio
AFTER INSERT ON giudizi
FOR EACH ROW
BEGIN
  DECLARE v_tot_revisori INT;
  DECLARE v_tot_giudizi  INT;
  DECLARE v_respingimenti INT;

  -- Quanti revisori sono assegnati?
  SELECT COUNT(*) INTO v_tot_revisori
  FROM revisioni
  WHERE id_bilancio = NEW.id_bilancio;

  -- Quanti hanno gia' votato?
  SELECT COUNT(*) INTO v_tot_giudizi
  FROM giudizi
  WHERE id_bilancio = NEW.id_bilancio;

  -- Se tutti hanno votato...
  IF v_tot_giudizi = v_tot_revisori THEN
    SELECT COUNT(*) INTO v_respingimenti
    FROM giudizi
    WHERE id_bilancio = NEW.id_bilancio
      AND esito = 'respingimento';

    IF v_respingimenti > 0 THEN
      UPDATE bilanci
      SET stato = 'respinto'
      WHERE id = NEW.id_bilancio;
    ELSE
      UPDATE bilanci
      SET stato = 'approvato'
      WHERE id = NEW.id_bilancio;
    END IF;
  END IF;

  -- Ricalcola affidabilita'
  CALL sp_aggiorna_indice_affidabilita(
    NEW.username_revisore);
END$$

SP14 — Indice di affidabilita'

CREATE PROCEDURE
  sp_aggiorna_indice_affidabilita(
    IN p_username VARCHAR(50)
)
BEGIN
  DECLARE v_totale INT DEFAULT 0;
  DECLARE v_positivi INT DEFAULT 0;
  DECLARE v_indice DECIMAL(3,2)
    DEFAULT 0.00;

  -- Giudizi totali del revisore
  SELECT COUNT(*) INTO v_totale
  FROM giudizi
  WHERE username_revisore = p_username;

  -- Giudizi con approvazione pura
  SELECT COUNT(*) INTO v_positivi
  FROM giudizi
  WHERE username_revisore = p_username
    AND esito = 'approvazione';

  -- Calcolo: positivi / totale
  IF v_totale > 0 THEN
    SET v_indice =
      ROUND(v_positivi / v_totale, 2);
  END IF;

  UPDATE revisori
  SET indice_affidabilita = v_indice
  WHERE username = p_username;
END$$

Esempio: se un revisore ha emesso 8 giudizi di cui 6 "approvazione" pura, il suo indice sara' 6/8 = 0.75. I giudizi "approvazione_con_rilievi" non contano come positivi.

Codice SQL: Registrazione Atomica + DDL

SP2 — sp_registra_utente (transazione)

CREATE PROCEDURE sp_registra_utente(
  IN p_username      VARCHAR(50),
  IN p_password_hash VARCHAR(255),
  IN p_codice_fiscale CHAR(16),
  IN p_data_nascita  DATE,
  IN p_luogo_nascita VARCHAR(100),
  IN p_ruolo         VARCHAR(20),
  IN p_email         VARCHAR(150),
  IN p_curriculum    VARCHAR(255)
)
BEGIN
  -- Se fallisce qualcosa: ROLLBACK
  DECLARE EXIT HANDLER FOR SQLEXCEPTION
  BEGIN
    ROLLBACK;
    RESIGNAL;
  END;

  START TRANSACTION;

  -- 1) Tabella padre
  INSERT INTO utenti (username,
    password_hash, codice_fiscale,
    data_nascita, luogo_nascita, ruolo)
  VALUES (p_username, p_password_hash,
    p_codice_fiscale, p_data_nascita,
    p_luogo_nascita, p_ruolo);

  -- 2) Email
  INSERT INTO email_utente
    (username, email)
  VALUES (p_username, p_email);

  -- 3) Sotto-tabella per ruolo
  IF p_ruolo = 'revisore' THEN
    INSERT INTO revisori (username)
    VALUES (p_username);
  ELSEIF p_ruolo = 'responsabile' THEN
    INSERT INTO responsabili
      (username, curriculum_pdf)
    VALUES (p_username, p_curriculum);
  END IF;

  COMMIT;
END$$

Le 3 INSERT sono atomiche: se una fallisce (es. username duplicato), nessuna riga viene inserita.

DDL — Esempio tabelle con vincoli

-- Gerarchia utenti: tabella padre
CREATE TABLE utenti (
  username      VARCHAR(50)  NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  codice_fiscale CHAR(16)   NOT NULL,
  data_nascita  DATE         NOT NULL,
  luogo_nascita VARCHAR(100) NOT NULL,
  ruolo ENUM('amministratore',
    'revisore','responsabile') NOT NULL,
  PRIMARY KEY (username),
  UNIQUE KEY (codice_fiscale)
) ENGINE=InnoDB;

-- Sotto-tabella revisori (stessa PK)
CREATE TABLE revisori (
  username VARCHAR(50) NOT NULL,
  nr_revisioni INT DEFAULT 0,
  indice_affidabilita DECIMAL(3,2)
    DEFAULT 0.00,
  PRIMARY KEY (username),
  FOREIGN KEY (username)
    REFERENCES utenti(username)
    ON DELETE CASCADE
) ENGINE=InnoDB;

-- Competenze con CHECK
CREATE TABLE competenze_revisore (
  username VARCHAR(50) NOT NULL,
  nome_competenza VARCHAR(100) NOT NULL,
  livello TINYINT NOT NULL,
  PRIMARY KEY (username, nome_competenza),
  FOREIGN KEY (username)
    REFERENCES revisori(username)
    ON DELETE CASCADE,
  CONSTRAINT chk_livello
    CHECK (livello BETWEEN 0 AND 5)
) ENGINE=InnoDB;

Wrapper DB (includes/db.php)

function callSP($sp_name, $params=[]) {
  $pdo = getDBConnection();
  $placeholders = implode(',',
    array_fill(0, count($params), '?'));
  $sql = "CALL {$sp_name}({$placeholders})";
  $stmt = $pdo->prepare($sql);
  $stmt->execute($params);
  $results = $stmt->fetchAll();
  $stmt->closeCursor();
  return $results;
}

Catena CASCADE (integrita' referenziale)

Tutte le FK usano ON DELETE CASCADE. Ecco cosa succede se cancello un utente:

DELETE FROM utenti WHERE username='mario'

→ CASCADE elimina email_utente (le sue email)
→ CASCADE elimina revisori (la riga revisore)
 → CASCADE elimina competenze_revisore
 → CASCADE elimina revisioni (assegnazioni)
 → CASCADE elimina note_revisione
 → CASCADE elimina giudizi

E se cancello un'azienda:

DELETE FROM aziende WHERE id=1

→ CASCADE elimina bilanci dell'azienda
 → CASCADE elimina valori_bilancio
 → CASCADE elimina voci_indicatori
 → CASCADE elimina revisioni
 → CASCADE elimina note_revisione
 → CASCADE elimina giudizi
(+ trigger T4 decrementa nr_bilanci)

function requireRole(string $ruolo): void {
